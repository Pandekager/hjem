<action>
1. Create `/etc/systemd/system/mcserver.service` using `sudo tee`. The service file:

```ini
[Unit]
Description=BMC4 Minecraft Forge 1.20.1 Server
After=network.target

[Service]
Type=simple
User=rasmus
WorkingDirectory=/home/rasmus/Games/MCServer/BMC4_Server_Pack_v57.5/
ExecStart=/usr/lib/jvm/java-17-openjdk/bin/java @user_jvm_args.txt @libraries/net/minecraftforge/forge/1.20.1-47.4.18/unix_args.txt nogui
Restart=on-failure
RestartSec=30
KillMode=process
TimeoutStartSec=120
TimeoutStopSec=60

[Install]
WantedBy=default.target
```

   - Per D-01: systemd service at `/etc/systemd/system/mcserver.service` — HOST-01
   - `User=rasmus` — matches current manual startup, not root
   - `ExecStart` uses Java 17 with Forge unix_args.txt (verified from disk, no assumptions)
   - `@user_jvm_args.txt` and `@libraries/...` are relative paths resolved via WorkingDirectory
   - `Type=simple` — Java process runs in foreground, systemd tracks PID
   - `KillMode=process` — kills the entire Java process tree on stop
   - `TimeoutStartSec=120` — Forge boot can take 60-90s
   - `TimeoutStopSec=60` — MC world save should complete within 60s
   - `Restart=on-failure` — auto-restart on crash but NOT on manual stop (systemd distinguishes)
   - Install section present but service will be DISABLED per HOST-02

2. Run `sudo systemctl daemon-reload` to register the new unit.

3. Verify the unit is recognized: `sudo systemctl cat mcserver` — must show the full unit file content. Confirm `ExecStart` line matches exactly.

4. Disable the service: `sudo systemctl disable mcserver` — ensures server stays off at boot (per HOST-02).

5. Verify it's disabled: `systemctl is-enabled mcserver` must print `disabled`.

6. Create a reference copy at `.planning/services/mcserver.service` with the exact same content (for documentation and future phases).
</action>

   ```bash
   sudo systemctl start mcserver
   ```
   Wait ~5 seconds for the Java process to initialize.

2. **Check status confirms running:**
   ```bash
   systemctl status mcserver
   ```
   Expected: Shows `Active: active (running)` since boot time, with Java process PID visible.
   Also check: `ps aux | grep java` shows a Java 17 process running the Forge server.

3. **Allow server to boot fully (optional but recommended):**
   Wait about 30-60 seconds, then check the logs:
   ```bash
   journalctl -u mcserver -n 20 --no-pager
   ```
   Expected output includes Forge loading mods, then "Done" or similar server-ready message.

4. **Stop the server:**
   ```bash
   sudo systemctl stop mcserver
   ```

5. **Confirm stop was clean:**
   ```bash
   systemctl status mcserver
   ```
   Expected: Shows `Active: inactive (dead)`.
   Also check: `ps aux | grep java` shows no minecraft java process.

6. **Verify off-by-default (HOST-02):**
   ```bash
   systemctl is-enabled mcserver
   ```
   Expected: `disabled`

If all 6 steps pass, HOST-03 and HOST-04 are satisfied — server can be started and stopped manually.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| systemd unit → Java process | systemd manages the MC server as a child process via cgroups. Correct ExecStart arguments are critical. |
| sudo access → service control | Managing mcserver.service requires sudo (host-level control). Only trusted user (rasmus) has sudo access. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-001 | Tampering | ExecStart command arguments | mitigate | Verify ExecStart line matches verified Java 17 path + relative @-args before starting |
| T-01-002 | Denial of Service | Java OOM (8-10GB RAM) | accept | If host runs out of memory, OOM-killer terminates Java; service restarts via Restart=on-failure (30s delay). Acceptable for a home server — user can monitor RAM usage manually |
| T-01-003 | Elevation of Privilege | service runs as User=rasmus | mitigate | Systemd drops privileges from root to rasmus before launching Java. Reduces blast radius if Java is compromised. No root needed after socket activation |
| T-01-004 | Spoofing | Service file integrity | accept | Only root can write to /etc/systemd/system/. Single-user system (rasmus has sudo). Risk accepted as low-value target on private LAN |
</threat_model>
